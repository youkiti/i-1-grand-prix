# 用語解説（Glossary of Terms）

| 用語 | 解説（高校生にもわかる説明） |
|------|------------------------------|
| **船荷証券 (Bill of Lading / B/L)** | 船で荷物を運ぶ際に発行される「預かり証」兼「引換券」です。これを持っている人だけが荷物を受け取れるため、紙自体に価値がある重要な書類です。 |
| **MLETR (電子譲渡可能記録に関するUNCITRALモデル法)** | 国連が定めた「電子データを紙の書類と同じように扱えるようにするための国際的なルールの手本」です。日本もこのルールに合わせて法整備を進めています。 |
| **電子船荷証券記録 (Electronic Bill of Lading Record)** | 紙の船荷証券を電子データ化したものです。今回の法改正で、紙と同じ法的効力を持たせるための名称として採用されました。 |
| **支配 (Control)** | 紙の書類を「持っている（占有する）」状態を、電子データの世界で置き換えた言葉です。システム上で「自分だけが使える状態」にあることを指します。 |
| **電子裏書 (Electronic Endorsement)** | 紙の書類の裏に名前を書いて権利を譲る「裏書」の電子版です。電子署名を使って、権利を別の人に移す手続きのことです。 |
| **UCP600 (信用状統一規則)** | 貿易のお金のやり取り（信用状取引）に関する国際的な銀行のルールです。電子船荷証券が銀行でスムーズに扱われるために重要な基準です。 |
| **BCP (事業継続計画)** | 災害やシステム障害が起きたときに、どうやって仕事を続けるか、あるいは復旧させるかを決めた計画のことです。 |

# 1. パブリックコメントによる仮説の検証

審議会での議論の方向性が、パブリックコメントによって実務的な観点から支持・補強された論点です。

### 1. 国際標準（MLETR）への準拠と独自ルールの回避
審議会では、国際的な調和を重視し、MLETR（国連モデル法）と親和性の高い制度設計（B案/丙案）を採用する方針が示されました。パブリックコメントでも、日本独自の仕様（ガラパゴス化）を避け、国際標準に合わせるべきとの意見が多数寄せられ、審議会の方針が強く支持されています。

> 「MLETR、シンガポール法、Law Commission 草案がその記録の方法として磁気的方式が取られることを禁止している又は想定していないのか否かは定かではないものの、このような国際動向を踏まえれば、我が国における電子化された船荷証券の法律上の名称としても、無用な誤解を避ける趣旨で、「電子」を用いるべき。」 [出典: doc_004, p.30]

> 「日本独自のシステムは無意味。つくるなら「世界共通（含む後進国）」にして。国際的な標準化に向けた取り組みや、後進国も含めた共通システムの構築に主導的な役割を果たすべきだ。」 [パブコメ: 6b907a94-6a9a-4209-a7b8-122deb9ec5f2]

### 2. 業務効率化とコスト削減への期待
審議会では、システム導入の負担を考慮しつつもデジタルファーストを志向する議論が行われました。パブリックコメントでは、現状の紙ベースの手続きによる遅延（特に近隣国向け）や管理コストが現場の大きな負担となっており、電子化による効率化が切実に求められていることが確認されました。

> 「デジタルファーストを志向すべきであるとの考え方を重視すると、運送人又は船長にそのような義務を負わせることも考えられる」 [出典: doc_004, p.1]

> 「船足が短い地域、例えば韓国向けなどは、現場書類より荷物の方が先に届いてしまっているので、電子化になれば、書類が遅れて痛感ができないと言う。現在の問題が解決されると思う。」 [パブコメ: fc0f1f96-af44-4a56-bc05-b539824e9442]

### 3. 国の認証機関制度の不採用（UKモデルの支持）
審議会では、国際的な利用を阻害しないよう、国の認証を受けた機関による関与を必須としない（イギリス型）方針が示されました。パブリックコメントでも、過度な規制や日本独自の認証制度が参入障壁となることを懸念し、国際的な相互運用性を重視する意見が見られます。

> 「電磁的船荷証券記録は、紙の船荷証券と同様に、国際的に利用されるものであることが想定されるところ、国の認証を受けた機関による関与を必要的なものとしてしまうと、かえって利用がされなくなるおそれがある。」 [出典: doc_004, p.18]

> 「各国のスタンダードと早期に合わせるべき。」 [パブコメ: b8471700-fa9a-48a1-8501-b44d9cad318f]

# 2. パブリックコメントによる反対・懸念の顕在化

審議会での議論に対し、パブリックコメントで懸念が示されたり、議論の方向性と実務感覚に乖離が見られる論点です。

### 1. システムの信頼性とセキュリティに対する強い懸念
審議会では、システム上の些細な問題で法的効力が争われるのを避けるため、一般的な「信頼性（Reliability）」を有効要件とせず、訓示規定（丙案）に留める方針が優勢でした。しかし、パブリックコメントでは、サイバー攻撃やシステム障害による業務停止、データ改ざんへの懸念が極めて強く、法的な担保や厳格なセキュリティ基準を求める声が目立ちます。

> 「電子船荷証券記録に対する一定の信頼性が求められることは当然の前提ではあるものの、これを独立した電子船荷証券記録の有効要件とすると、その取引においては特に問題がなかったにもかかわらず、システム上の些細な問題点を巡って後に争いが生じるおそれがある。」 [出典: doc_006, p.16]

> 「昨今、国際的な大企業を狙ったサイバー攻撃が頻発しています。紙媒体を廃止してしまうと、このような事態に対応できません。」 [パブコメ: 1dab198b-8eb4-4193-8d68-8dc10abdc510]

### 2. システム障害時のバックアップ（紙への回帰）の必要性
審議会では、運送人に対する一方的な「紙への転換請求権」を認めない（X案/合意ベース）方針が採用されました。しかし、パブリックコメントでは、システム障害時（AWS障害等の事例）に物流が止まるリスクが指摘されており、緊急時の紙媒体への回帰手段（バックアップ）としての転換機能の重要性が示唆されています。

> 「デジタルファーストの志向からすると紙の船荷証券への転換請求権は認めるべきではない...運送人にとって酷である場面も想定される」 [出典: doc_006, p.24]

> 「1分1秒も無駄にしない、1分でも早くスムーズに更新される正確なシステムが出来ない限りは導入すべきじゃない。」 [パブコメ: d33788b3-9d37-43f6-8d94-d796328b7f45]
> （注：システム障害時のリスクに対する強い懸念の表れ）

### 3. 導入スピードに対する温度差
審議会では、法体系との整合性や国際動向を見極めながら慎重に検討が進められていますが、パブリックコメントでは「周回遅れ」であるとの認識が強く、国際競争力維持のために一刻も早い導入を求める声が多数を占めています。

> 「我が国の法体系に大きな影響を及ぼすこととなるため、その採否については慎重に検討すべきであると考えられる」 [出典: doc_001, p.1]

> 「ぱっと見の印象では、もう10年も20年も前には実装していないと遅すぎるくらいなのかなと感じました。」 [パブコメ: c8853765-bb12-4aa0-883b-e4fe9149fd00]

# 3. 新たなインサイト・論点

審議会資料には明示的に現れていなかったが、パブリックコメントから得られた新たな視点です。

### 1. 経済安全保障とデータ主権
データの保管場所や管理主体に関する懸念が示されました。特に、特定国（仮想敵国等）によるデータ管理や、不正な目的を持った国による密輸リスクなど、経済安全保障の観点からのシステム要件が求められています。

> 「また特定国の管理だけでなく国際的に信用できる体制でデータ管理しないと不正な目的をもった国であれば密輸や裏ルートでの輸出入が可能になってしまう。」 [パブコメ: 11b917fc-e64b-4d4a-a520-6af11586a56a]

### 2. UCP600（信用状統一規則）との明示的な連携
法律上の効力だけでなく、銀行実務における決済ルール（UCP600）における電子B/Lの取り扱いを明確化してほしいという具体的な要望が出されています。これは法務省の管轄外かもしれませんが、実務上不可欠な要素です。

> 「ICC荷為替信用状に関する統一規則及び慣習（UCP600)に、電子BLの取り扱いを明記すべき。これは絶対絶対必要！」 [パブコメ: 5e581146-cc79-40e5-8c4b-2ee1f79fd30d]

### 3. AI・自動化との親和性
単なるペーパーレス化だけでなく、AIによるデータ入力の自動化やデータサイエンスへの活用を見据えた規格統一（フォーマットの標準化）への期待が寄せられています。

> 「電子化し標準化することで、未来に向かって作業効率化のデータサイエンスができそう [...] 作業自体の自動化も可能かと思います」 [パブコメ: 842ed3cb-cd56-4cea-b946-830096697321]

# 4. 対立軸に対するパブリックコメントの反応

### 技術的要件としての信頼性の要件（審議会: 信頼性要件甲案（規律しない）/乙案（有効要件）/丙案（訓示規定））

| パブコメの立場 | 代表的な意見 | 出典 |
| :--- | :--- | :--- |
| **信頼性重視（乙案寄り）** | 「電子データの本物確認、二重三重のバックアップ、改ざん防止、暗号機能の確保が可能か。...国際的に信用できる体制でデータ管理しないと」<br>「昨今、国際的な大企業を狙ったサイバー攻撃が頻発しています。紙媒体を廃止してしまうと、このような事態に対応できません。」 | `[パブコメ: 11b917fc-e64b-4d4a-a520-6af11586a56a]`<br>`[パブコメ: 1dab198b-8eb4-4193-8d68-8dc10abdc510]` |
| **イノベーション重視（甲/丙案寄り）** | 「他国の成果と課題を踏まえてより良い形で運用開始してくれれば良い。」<br>（注：直接的に「要件を緩くせよ」という意見は少ないが、国際標準への準拠を求める声はこれに近い） | `[パブコメ: aef81ac9-70f8-4a1a-b4b3-452d9911c735]` |

- **審議会での状況**: 継続検討（丙案：訓示規定が優勢）
- **パブコメの傾向**: セキュリティリスクへの懸念が非常に強く、実質的には「乙案（有効要件化）」や厳格な基準を求める声が多い。ただし、法的な有効要件とすべきか、技術的なガイドラインで対応すべきかの区別は明確ではない。
- **合意形成への示唆**: 法律上は丙案（訓示規定）として柔軟性を残しつつ、**省令やガイドラインで高度なセキュリティ基準（BCP含む）を具体的に定める**ことで、パブコメの懸念を解消する方向性が望ましい。

### 転換請求権の要否（審議会: X案（否定/合意のみ） vs Y案（肯定/義務化））

| パブコメの立場 | 代表的な意見 | 出典 |
| :--- | :--- | :--- |
| **Y案支持（義務化/バックアップ）** | 「システム障害対策も含めた運用をしているようです。」（他国事例への言及）<br>「1分1秒も無駄にしない...正確なシステムが出来ない限りは導入すべきじゃない。」（システム不全への懸念） | `[パブコメ: 47209dcc-5f65-44b5-8067-5c8c7dd63510]`<br>`[パブコメ: d33788b3-9d37-43f6-8d94-d796328b7f45]` |
| **X案支持（合意のみ/デジタル推進）** | 「紙での手続きはメリットがないので、電子手続きの方を標準にして欲しい。」 | `[パブコメ: 1488dc8d-02a7-4626-a9a8-6422d8ded5eb]` |

- **審議会での状況**: 決着（採用: X案/転換義務なし）
- **パブコメの傾向**: 明示的に「転換請求権」という言葉を使った意見は少ないが、システム障害時のリスクヘッジを求める声は強い。これは実質的に、緊急時の紙媒体への回帰（転換）を担保してほしいというニーズと解釈できる。
- **合意形成への示唆**: 運送人への一方的な転換義務（Y案）は課さないものの、**システム障害時等の緊急時における代替手段（紙への転換含む）の確保を、システム提供者や運送人の努力義務やBCPガイドラインとして推奨する**ことが現実的である。

# 5. 今後の対応方針案（国会アクションカード形式）

## 5.1 まとめ

本件は、国際競争力維持の観点から早期の法制化が強く望まれています。審議会の方針（MLETR準拠・機能的同等性）は概ね支持されていますが、**「セキュリティ・信頼性」への懸念**がパブコメで顕著です。
法律自体は柔軟な設計（訓示規定等）としつつ、運用面での安全性確保を政府に強く求めるアプローチが有効です。
**推奨コンボ**: カード2（委員会質疑）→ カード1（賛成討論）

## 5.2 論点マップ

| 論点 | 何が問題か | だれに影響か | どのレイヤーで動くか |
| :--- | :--- | :--- | :--- |
| **セキュリティ確保** | 法的要件を緩くすると、サイバー攻撃等で物流が止まるリスクがある。 | 荷主、運送人、消費者 | 政省令 / 監督強化 |
| **国際標準への適合** | 日本独自ルール（ガラパゴス化）になると、海外で通用しない。 | 輸出入企業 | 法律 / 条約 |
| **システム障害対策** | AWS障害等で電子記録が使えない時のバックアップがない。 | 物流現場 | ガイドライン / 民間契約 |
| **経済安全保障** | データが特定国に管理されると、物流情報が筒抜けになる。 | 国家安全保障 | 法律 / 運用監視 |

## 5.3 推奨アクション（カード別）

### カード2: 国会の各種委員会での質疑・発言

- **ねらい**: 法案の早期成立を後押ししつつ、パブコメで示された「セキュリティ不安」に対する政府の具体的な対応策を引き出す。
- **使うべき論点**: 「サイバーセキュリティ、データ信頼性」「経済安全保障」「システム障害時のBCP」
- **具体アクション**:
    - 法務委員会または国土交通委員会において、以下の点を質問する。
    - 「パブコメではサイバー攻撃やシステム障害への懸念が多数寄せられている。法律上は『信頼性』を訓示規定としているが、実務上の安全性はどう担保するのか？」
    - 「経済安全保障の観点から、データの保管場所や管理主体の適格性をどうチェックするのか？」
- **期待アウトカム**:
    - 主務大臣から、省令やガイドラインにおいてセキュリティ基準を明確化する旨の答弁を引き出す。
    - システム障害時の対応指針（紙への転換運用の推奨など）が示される。
- **リスクと代替策**:
    - リスク: 議論が細かくなりすぎて審議が遅れる。
    - 代替策: 附帯決議案として「セキュリティ確保と国際標準の維持」を盛り込む。

### カード1: 法律案・条約・予算に対する賛否の投票

- **ねらい**: 国際競争力維持のため、法案を速やかに成立させる。
- **使うべき論点**: 「国際競争力維持」「業務効率化」「国際標準（MLETR）への準拠」
- **具体アクション**:
    - 本会議および委員会での採決において「賛成」する。
    - 賛成討論において、「周回遅れを取り戻すための迅速な施行」と「現場の混乱を防ぐための丁寧な周知」を要望する。
- **期待アウトカム**: 法案成立による電子船荷証券の利用開始。
- **リスクと代替策**: 特になし（パブコメでも賛成多数のため）。
